# Сервис таблиц и форм

Веб-прототип для хранения, редактирования, импорта, экспорта и анализа таблиц и форм. По возможностям проект напоминает связку Google Таблиц и Google Форм: пользователь может создавать таблицы и формы, заполнять формы, смотреть ответы, делиться ресурсами и фильтровать данные.

Данные хранятся в ArangoDB. Приложение запускается через Docker Compose и отдаёт статический frontend с Node.js backend.

## Быстрый старт

```bash
docker compose up -d --build
```

После старта приложение доступно по адресу:

```text
http://127.0.0.1:8081
```

Если менялись JS/CSS/HTML-файлы, нужен rebuild образа:

```bash
docker compose up -d --build app
```

Обычный `docker compose restart app` только перезапускает уже собранный контейнер и не подтягивает изменения из рабочей папки.

## Тестовые пользователи

| Роль | Логин | Пароль |
|------|-------|--------|
| Администратор | `admin` | `admin` |
| Аналитик | `analyst` | `analyst` |
| Редактор | `editor` | `editor` |

Регистрация также доступна через интерфейс. Пароли тестовых пользователей в seed-данных хранятся как scrypt-хэши. Для боевого стенда замените тестовые учётные записи, пароль ArangoDB и cookie-настройки.

## Реализованные сценарии

- Авторизация и регистрация пользователей.
- Просмотр списков таблиц и форм.
- Создание, редактирование и удаление таблиц.
- Excel-подобная сетка 40x40 с фильтрами по значениям столбцов.
- Создание, редактирование и удаление форм.
- Предпросмотр форм и сохранение ответов.
- Разграничение доступа: владелец видит свои ресурсы, приглашённый пользователь видит ресурсы с выданным доступом.
- Журналы действий по таблицам и формам.
- Многокритериальная фильтрация списков и журналов.
- Импорт и экспорт данных в JSON, CSV, XML, XLSX и ODS.
- Статистика по метаданным таблиц/форм и по строкам выбранной таблицы.

## Backend и API

Backend находится в `src/server.js`.

Он обслуживает:

- статические файлы из `public/`;
- `POST /api/auth/login` — вход;
- `POST /api/auth/register` — регистрация;
- `GET /api/auth/me` — текущая сессия;
- `POST /api/auth/logout` — выход;
- `GET /api/state` — загрузка доступного текущему пользователю состояния;
- `PUT /api/state` — сохранение состояния;
- `POST /api/seed/reset` — сброс БД к seed-данным, доступен только администратору.

Сессия хранится в HttpOnly-cookie. Для изменяющих запросов используется CSRF-токен. Сервер не отдаёт password-хэши в клиентское состояние.

## Данные

Целевая структура БД описана в `docs/DataModel.md`.

Используемые коллекции ArangoDB:

- `users`
- `spreadsheets`
- `table_rows`
- `forms`
- `form_responses`
- `activity_log`
- `user_assets`
- `form_targets`

При первом запуске backend создаёт БД и коллекции, затем загружает `data/seed-data.json`. Если volume ArangoDB уже содержит данные, seed повторно не затирает существующую БД.

## Импорт и экспорт

Поддерживаемые форматы:

- JSON
- CSV
- XML
- XLSX
- ODS

XLSX и ODS внутри являются ZIP-архивами с XML-документами. Для работы с ними в браузере используется vendored библиотека `public/vendor/fflate.js`.

Заметка по версии, источнику и лицензии находится рядом: `public/vendor/fflate.NOTICE.md`.

## Безопасность

### Модель безопасности проекта

Пользовательскими данными считаются названия таблиц и форм, комментарии, ячейки, поля форм, варианты ответов, ответы пользователей, журналы действий и импортированные файлы. Чувствительными данными считаются пароли и сессионные cookie; сервер не отдаёт password-хэши, соли и идентификаторы сессий во frontend.

У таблиц и форм есть владелец (`ownerLogin`). Таблицу видит владелец и приглашённые пользователи. Приглашённый пользователь может редактировать содержимое таблицы, но не может изменить владельца, список приглашённых или удалить таблицу через отсутствие ресурса в payload. Форму видит владелец и приглашённые пользователи; приглашённый пользователь может открыть форму и отправить ответ, но не может менять структуру формы. Ответы формы видит владелец формы, а отправитель видит только собственные ответы.

### Реализованные меры защиты

- Пароли хранятся как scrypt-хэши с солью; seed-пользователи мигрируются из legacy-формата при старте.
- Сессия хранится на сервере, идентификатор сессии передаётся в `HttpOnly; SameSite=Lax` cookie.
- Для `PUT /api/state`, `POST /api/auth/logout` и `POST /api/seed/reset` требуется CSRF-токен.
- На вход и регистрацию действует базовая проверка логина, минимальная длина пароля и rate limit на неуспешные попытки входа.
- Все операции чтения и записи состояния проходят через backend; сервер фильтрует таблицы, формы, ответы и журналы по текущему пользователю.
- `ownerLogin`, `ownerId`, `role`, `isAdmin`, список пользователей и password-поля нельзя изменить через импорт или `PUT /api/state`.
- Владелец ресурса сохраняется сервером из существующего документа, а не из client payload.
- Приглашения нормализуются: владелец не добавляется в invited list, дубликаты удаляются, неизвестные пользователи отбрасываются.
- Пользовательский ввод перед вставкой в HTML экранируется через `escapeHtml`; текстовые значения вставляются через `textContent` там, где это возможно.
- Backend нормализует импортированные таблицы, формы, ответы и логи: ограничены размеры массивов, длины строк, типы полей и допустимые ссылки на формы/таблицы.
- JSON body ограничен 15 МБ, импортируемые файлы во frontend ограничены 10 МБ, XLSX/ODS дополнительно ограничены 25 МБ после распаковки.
- XLSX/ODS экспорт защищён от formula injection: значения, начинающиеся с `=`, `+`, `@`, опасного `-` или управляющих переводов строк, выгружаются как текст.
- Служебный payload в XLSX/ODS кодируется, чтобы скрытый лист не превращался в исполняемые spreadsheet-формулы.
- AQL-запросы к ArangoDB используют bind-переменные и фиксированные имена коллекций.
- Static server отдаёт только whitelist файлов из `public/`, без произвольной раздачи путей.
- Добавлены CSP, `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`.
- Ошибки backend не раскрывают внутренние stack trace пользователю.
- Секреты вынесены в переменные окружения; пример находится в `.env.example`.

### Ограничения текущей реализации

Это учебный прототип. Frontend всё ещё кеширует часть состояния в `localStorage`, поэтому данные в браузере нельзя считать доверенной границей безопасности: пользователь технически может изменить локальный cache. Реальная защита строится на серверных проверках при чтении и сохранении.

Для production недостаточно хранить данные только в frontend/localStorage и нельзя размещать настоящие секреты во frontend-коде. Тестовые пользователи и пароль ArangoDB из `docker-compose.yml` предназначены только для локального стенда; перед публикацией их нужно заменить через `.env`.

### Рекомендации для production

- Использовать HTTPS и включить `APP_SECURE_COOKIES=1`.
- Хранить пароли только в стойких хэшах с солью и настроить политику смены тестовых паролей.
- Использовать `HttpOnly`, `SameSite`, `Secure` cookie или строго продуманную JWT-стратегию.
- Добавить полноценную CSRF-защиту для всех cookie-based mutating endpoints.
- Добавить rate limiting не только на вход, но и на импорт, экспорт, создание ответов и регистрацию.
- Оставить server-side validation единственным источником доверия.
- Добавить аудит действий владельца и приглашённых редакторов.
- Настроить регулярные резервные копии ArangoDB.
- Настроить production CSP, security headers и безопасный CORS под реальные домены.
- Не коммитить `.env` с секретами, API-ключами или production-паролями.

### Как проверить безопасность вручную

- Войти двумя пользователями и попробовать открыть чужую таблицу/форму по ID.
- Попробовать изменить `ownerLogin`, `ownerId`, `role`, `isAdmin` через импорт или `PUT /api/state`.
- Пригласить пользователя в таблицу и проверить, что он редактирует ячейки, но не меняет владельца и приглашённых.
- Пригласить пользователя в форму и проверить, что он может отправить ответ, но не редактирует поля формы.
- Вставить `<script>alert(1)</script>` в название таблицы, ячейку, название формы, описание и ответ формы.
- Импортировать файл с опасными значениями и слишком длинными строками.
- Экспортировать XLSX/ODS со значениями, начинающимися с `=`, `+`, `-`, `@`, и проверить, что они открываются как текст.
- Проверить, что приглашённый пользователь формы не видит чужие ответы.

## Полезные команды

Проверить синтаксис JavaScript:

```bash
node --check src/server.js
node --check public/app.js
node --check public/data.js
node --check public/spreadsheet.js
```

Запустить security smoke checks против поднятого приложения:

```bash
npm test
```

Запустить локально без Docker, если ArangoDB доступна:

```bash
npm start
```

Пересобрать только приложение, не трогая volume БД:

```bash
docker compose up -d --build app
```

Полностью пересоздать контейнеры:

```bash
docker compose down
docker compose up -d --build
```
